## 작업 내역

### 1. [RegScan] 브리핑 파이프라인 V2 설계 확정

#### 배경

4/15 배치 파이프라인 실행 결과, unified briefing에서 "브리핑 본문은 '없음'이지만..." 생성.
근본 원인: LLM이 팩트 생성 + 기사 작성을 동시에 하는 구조적 결함.
가드레일/프롬프팅으로 해결 불가 — 파이프라인 재설계 필요.

#### 확정된 4-Step 구조

```
Step 1: 코드 → 개별 팩트카드 JSON (LLM 없음)
Step 2: 팩트카드 N개 → LLM → 트렌드/교차분석
Step 3: 트렌드 + 팩트카드 → LLM → 최종 기사
Step 4: 코드 → Validator → 사실/표현 검증 (LLM 없음)
```

#### 핵심 설계 요소

- **ALLOWED_PHRASES 매트릭스**: (상태 × 신뢰도) → 고정 문구. LLM이 팩트를 생성하지 않음
- **hira_evidence_type**: NON_REIMBURSED/NOT_FOUND 판정 근거 추적 (exact_price_match / board_missing / bridge_unmatched / source_not_loaded)
- **Validator 2단계 체크**: 하드(날짜/가격/상태 exact) + 소프트(문장 semantic 허용)
- **자동 교정 금지**: 위반 시 문단 폐기 → fallback fact-only 삽입 → 로그. silent corruption 방지

#### 설계 문서

`docs/design/briefing_pipeline_v2.md` — 전체 스키마, 데이터 흐름, 마이그레이션 계획

### 3. [RegScan] Step 1 구현 완료 — `fact_card.py` (P0)

#### 기존 소스와의 차이점

| 항목 | 기존 (`briefing.py`) | 신규 (`fact_card.py`) |
|------|---------------------|---------------------|
| **팩트 생성 주체** | LLM이 drug dict를 보고 문장 생성 | **코드가 ALLOWED_PHRASES에서 선택** |
| **HIRA 상태 분기** | `_enrich_via_bridge()`에서 if/elif 4개로 fact 문자열 조합 | **HIRA_PHRASES 딕셔너리**: (상태, 신뢰도) → 고정 문구. 10개 조합 |
| **MFDS 판정** | `mfds_status == "미허가"`면 그대로 LLM에 전달 → LLM이 "미허가" 단정 | **MFDS_PHRASES**: 허가일 없으면 "허가 여부 추가 확인 필요" (단정 불가) |
| **FDA 시제** | few-shot으로 LLM에 위임 (hallucination 위험) | **FDA_PHRASES**: date vs today 비교 → "승인 완료"/"승인 예정" 코드 확정 |
| **가드레일** | `is_guardrailed` bool + `guardrail_note` 문자열 | **guardrail_notes 리스트**: HIRA/MFDS 각각 근거 명시 |
| **evidence_type** | 없음 | **hira_evidence_type**: exact_price_match / board_missing / bridge_unmatched / source_not_loaded |
| **출력 형태** | `hira_data` dict를 drug_dict에 주입 → `_extract_drug_intel()`이 20필드 추출 → LLM에 전달 | **FactCard dataclass**: `to_compact_dict()` (~10줄), `all_fact_phrases` (검증용), `hard_check_values` (날짜/가격) |
| **LLM 의존도** | 팩트 판정 + 기사 작성 동시 | **팩트 판정 0% LLM** — 코드가 전부 결정 |

#### 핵심 변화

1. **ALLOWED_PHRASES 매트릭스** — 모든 팩트 문구가 딕셔너리에서 나옴. 코드에 없는 문구는 나올 수 없음
2. **"미허가" 단정 제거** — 허가일 없는 "미허가"는 "허가 여부 추가 확인 필요"로 자동 완화
3. **hira_evidence_type** — NOT_FOUND / NON_REIMBURSED 판정 근거 추적 가능
4. **to_compact_dict()** — LLM 입력이 약물당 10줄로 압축 (기존: 20필드 raw dict)
5. **hard_check_values** — Validator(Step 4)에서 날짜/가격 exact match 검증용

| 약물 | MFDS (기존) | MFDS (신규) | 변화 |

| PEMBROLIZUMAB | 식약처 허가 | 식약처 허가 완료 (2015-03-25) | 날짜 추가 |
| BELUMOSUDIL | **식약처 미허가** | **허가 여부 추가 확인 필요** | 단정 → 확인 필요 |

### 8. [RegScan] MFDS Enrichment 리팩터 — 코드 기반 매칭으로 전환

#### 발견된 문제

`_enrich_with_mfds()`가 영문 INN을 `item_name`에 넣어 식약처 API 텍스트 검색 → **결과 0건, 완전히 작동하지 않는 코드**였음.
- 식약처 API `item_name`은 한글 품목명 전용 → 영문 INN 검색 불가
- 2026-02-04 실험(`docs/research/matching/`)에서 이미 코드 기반 매칭 경로가 검증되어 있었으나 적용되지 않음

#### AS IS → TO BE

| | AS IS | TO BE |
|--|-------|-------|
| 검색 방식 | `item_name=INN` (영문 텍스트) | `edi_code=제품코드` (9자리 코드) |
| 매칭 결과 | 0건 (작동 안 함) | 정확 1건 매칭 |
| 코드 출처 | 없음 | IngredientBridge → `raw_data["제품코드"]` |
| API 호출 | INN 수 × 0.2초 (무의미) | 제품코드 있는 것만 호출 |

#### 코드 체계 확인

- HIRA `제품코드` = MFDS `EDI_CODE` (동일 코드 체계)
- 식약처 API는 `edi_code` (소문자) 파라미터만 인식 (대문자 무시)
- 삭제된 구 코드나 OTC 약물은 MFDS에 EDI_CODE 미등록 → 스킵

### 9. [RegScan] 의약품 코드 체계 조사 — ATC 기반 매칭 검토

#### 현황: 각 소스별 보유 코드

| 소스 | 보유 코드 | ATC |
|------|----------|-----|
| FDA (openfda) | UNII, RxCUI, NDC | 없음 (RxNorm API 경유 필요) |
| EMA | EU number | 직접 제공 |
| 약가마스터 (HIRA) | 품목기준코드, 표준코드, 일반명코드 | `국제표준코드(ATC코드)` 필드 있음 |
| HIRA ATC 매핑 | 제품코드, 주성분코드 | ATC ↔ 제품코드 매핑 |
| MFDS | ITEM_SEQ, EDI_CODE, PRDLST_STDR_CODE | 없음 (약가마스터 경유) |

#### ATC 커버리지 문제

| 데이터 | ATC 있음 | ATC 없음 |
|--------|---------|---------|
| HIRA 약가 (50,271 제품코드) | 20,693 (41.2%) | 29,578 (58.8%) |
| 약가마스터 (305,522건) | 234,489 (76.8%) | 71,033 (23.2%) |

**ATC 단독으로는 절반도 못 잡음** → 복수 경로 fallback 필수

#### 이상적인 매칭 우선순위 (향후 구현)

```
1순위: ATC 코드 (FDA→RxNorm→ATC, EMA→ATC 직접)
2순위: 제품코드 = EDI_CODE (현재 MFDS 매칭에 사용 중)
3순위: 품목기준코드 경유 (약가마스터 PRDLST_STDR_CODE)
최후: INN 문자열 (IngredientBridge, 현재 HIRA 매칭에 사용 중)
```

#### 핵심 리소스

- RxNorm API (무료): https://rxnav.nlm.nih.gov/ — FDA RXCUI → ATC 변환
- HIRA ATC 매핑: `data/bridge/건강보험심사평가원_ATC코드_매핑_목록_20250630.csv`
- 약가마스터: `data/bridge/yakga_master_latest.csv` — ATC + 품목기준코드 + 일반명코드
- NDC→ATC 오픈소스: https://github.com/fabkury/ndc_map
