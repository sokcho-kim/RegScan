### KIPRIS 추가 서비스 (구현 보류)

KIPRIS에서 3개 서비스를 신청했으나, 2개는 지금 구현 우선순위가 낮다:

#### 분류코드 API (`ClassificationService`)

| 항목 | 내용 |
|------|------|
| 엔드포인트 | `http://plus.kipris.or.kr/openapi/rest/ClassificationService/searchIPCInfo` |
| 파라미터 | `ipcNumber` (IPC코드/단어), `docsStart` (페이지) |
| 용도 | IPC 코드의 한글/영문 설명 조회 |
| 활용 시점 | 특허 검색 결과 표시 시 IPC 코드 해석 (예: A61K → "의약용 제제") |
| 우선순위 | **낮음** — 유틸리티 성격, 특허 대시보드 구현 시 함께 |

#### 의약품 국제일반명칭 API (`INNService`)

| 항목 | 내용 |
|------|------|
| 엔드포인트 | `http://plus.kipris.or.kr/openapi/rest/INNService/basicINNInfo` |
| 오퍼레이션 | INN 기본조회, INN 명칭조회, 변동정보 |
| 데이터 | 51,621건 (1954~2019), 영어/라틴어/프랑스어 명칭 |
| 용도 | INN(국제일반명) 참조 사전 — IngredientBridge 보조 |
| 우선순위 | **낮음** — HIRA 약가마스터로 국내 INN 이미 커버, 매칭 정확도 이슈 시 활용 |
| 한계 | 2019년까지만 수록, 반기 업데이트 (최신 약물 미포함) |

**결론:** 둘 다 코드는 안 짜도 되고, 필요할 때 KIPRIS `getWordSearch`와 동일 패턴으로 구현하면 된다. 엔드포인트/파라미터는 이 일지에 기록 완료.

#### 실행 전략

```
1단계: oncology 단일 스트림 (안전)

2단계: 결과 검증
   - output/stream_results/ JSON 확인
   - DB drugs/events 건수 확인
   - Step 4.8 보조 인텔리전스 건수 확인
   - 브리핑 생성 여부 확인

3단계: 전체 3-stream (검증 후)
   python -m regscan.batch.pipeline
```

#### 검증 체크리스트

| 단계 | 검증 | 성공 기준 |
|------|------|----------|
| DB 초기화 | `init_db` | OK |
| 스트림 수집 | therapeutic/oncology | 10+건 약물 |
| 결과 병합 | GlobalRegulatoryStatus | 생성 확인 |
| DB 적재 | drugs/events upsert | 건수 > 0 |
| KHIDI/KDCA | 보도자료 수집 | OK or graceful skip |
| 보조 인텔리전스 | PMDA/NICE/MFDS/MOHW/국회/DART/KIPRIS | 각각 건수 기록 |
| 브리핑 생성 | LLM 호출 | 1건 이상 생성 |
| JSON 저장 | stream_results/ | 파일 생성 |

#### 사전 준비

- DB 백업: `cp data/regscan.db data/regscan.db.bak`
- LLM 키 확인: OPENAI/GEMINI 중 최소 1개
- 디스크 여유: 최소 5GB

#### 소요 시간: ~30분 (oncology만), ~2시간 (전체 + 브리핑)
