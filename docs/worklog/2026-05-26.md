## 1. 식약처(MFDS) 일일스캔 0건 — 원인 규명

증상: daily_scan의 `mfds_new`가 5-20·5-21·5-26 전부 0인데 `errors`는 비어 있음. FDA/EMA는 정상(73·86건).

### 근본 원인 (세 겹)

1. **데이터 신선도** — `_scan_mfds`가 읽는 `data/mfds/permits_full_*.json`이 2026-02-03 스냅샷에 멈춤. 최신 허가일 2026-02-02. 오늘 기준 최근 N일 필터를 통과하는 게 0건.

2. **구조적 단절** — 갱신용 `MFDSPermitWorker`는 수집분을 PostgreSQL(`regulatory_events`)에만 upsert. 파일을 읽는 `_scan_mfds`와 경로가 연결된 적이 없음. 워커가 돌아도 스캔이 읽는 파일은 안 바뀜.

3. **delta 로직 버그** — 워커 delta 모드는 "API가 최신순 반환"을 가정하고 오래된 허가일이 나오면 중단. 그러나 공공데이터포털 허가정보 API는 **무작위 순서**(1955, 1955, 1956...)로 반환 → 1페이지에서 즉시 중단 → ~0건.

4. (조용한 실패) errors가 비어 멈춤을 인지 못 함. 워커 실패 로그도 `debug` 레벨이라 묻힘.

### API 제약 (재발방지 설계 근거)

- 날짜 필터 파라미터 없음 + 무작위 순서 + 허가 **발행이 ~4일 지연**(최신 5-26 아닌 5-22).
- 따라서 증분 수집 불가. **전체(약 4.3만건) 재덤프 후 로컬 필터**만 가능.
- 스캔 창은 발행지연보다 커야 함. `SCAN_DAYS_BACK=30`이라 정상. `days_back=1`이면 멀쩡해도 0건.

## 2. 수정 (재발방지)

- **`scripts/refresh_mfds_dump.py`** (신규) — 전체 재덤프 → `permits_full_YYYYMMDD.json`. 최근 3개만 유지, 수집 0건이면 기존 캐시 보호.
- **`daily_scanner._scan_mfds`** — 신선도 가드: 캐시가 (창+1)일 이상 묵으면 `result.errors`에 경고. 조용한 0건 방지. `_mfds_cache_age_days` 헬퍼 추가.
- **`scheduler.py` / `batch/pipeline.py`** — 깨진 delta 워커 호출을 전체 덤프 갱신으로 교체. 실패 숨기던 `debug` → `warning`.
- **`run_daily_scan.py`** — `--refresh-mfds` 플래그(스캔 전 덤프). cron 한 줄 운영용.
- **`.gitignore`** — `permits_full_*.json` 추적 제외(매일 재생성 32MB), 기존 덤프 untrack.

### 검증

- 라이브 API: 키 유효, 전체 43,240건.
- 재덤프 후 `_scan_mfds`: days_back=7 → 21건, 30 → 115건 (실제 약물명·허가일 확인).
- 신선도 가드: 묵은 캐시(112일) 경고 발동, 신선(0일) 정상.

### 운영 (온프렘 리눅스 cron)

```bash
0 8 * * * cd /srv/RegScan && .venv/bin/python scripts/run_daily_scan.py --days 30 --refresh-mfds --generate-briefing >> logs/daily_scan.log 2>&1
```

prod Docker는 `batch.pipeline` 3-stream 모드(`_enrich_with_mfds`는 별개 메커니즘). daily_scan의 mfds_new는 DailyScanner 경로(scheduler / run_daily_scan)만 생성하므로, 온프렘은 cron+run_daily_scan 방식으로 운영.
