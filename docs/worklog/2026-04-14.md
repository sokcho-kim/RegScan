### 1. [RegScan] IngredientBridge Decomposition 파이프라인 설계 + 구현 (P1 — 구현 완료)

#### 1-0. 배경

4/13 분석 결과 unmatched 882건(고유 성분명 기준)에 대해 Level 1~4 삭제 방식을 검토했으나,
**엔지니어링 정책에 의해 삭제 방식 전면 폐기**. 이유:

- 제형 suffix 삭제 시 약동학적 차이(= 약가 차이) 정보 소실
- 염 형태 삭제 시 다른 화합물을 동일 성분으로 오매칭 위험
- 함량/퍼센트는 Strength 컬럼으로 파싱해야 할 핵심 데이터
- 첫 단어 매칭은 데이터 오염도 과다

**새 방향**: 삭제(Deletion) → 분해(Decomposition). 원본 텍스트를 4개 컬럼으로 쪼개되 정보를 버리지 않는다.

#### 1-1. 엔지니어링 정책

| Level | 기존(폐기) | 신규(채택) |
|-------|-----------|-----------|
| 1 제형 | suffix 삭제 | Variant 속성으로 분리 → `formulation` 컬럼 |
| 2 염 | suffix 삭제 | `ref_normalization_map` 기반 표준명 치환 → `salt` 컬럼 |
| 3 함량 | 숫자/% 삭제 | 숫자/% 파싱 → `strength` 컬럼 |
| 4 첫단어 | 첫 단어 매칭 | **불허** (데이터 오염도 과다) |

#### 1-2. 출력 스키마: `DecomposedIngredient`

```python
@dataclass
class DecomposedIngredient:
    """성분명 분해 결과 — 원본 텍스트의 무손실 분해"""
    raw: str                    # 원본 그대로 ("Rivaroxaban Micronized")
    base_inn: str               # 핵심 INN ("rivaroxaban")
    salt: str | None            # 염 형태 ("hydrochloride", "sodium" 등)
    formulation: str | None     # 제형 variant ("micronized", "coated granules" 등)
    strength: str | None        # 함량 ("42%", "97%", "10.7mg" 등)
    match_confidence: float     # 분해 신뢰도 (0.0~1.0)
    
    # 계층 구조
    @property
    def variant_key(self) -> str:
        """Base + Salt + Formulation → 약가 매칭용 키"""
        parts = [self.base_inn]
        if self.salt:
            parts.append(self.salt)
        if self.formulation:
            parts.append(self.formulation)
        return " ".join(parts)
    
    @property
    def base_key(self) -> str:
        """Base INN만 → 성분 동일성 확인용 키"""
        return self.base_inn
```

**계층 관계**:
```
Base INN (성분 동일성)
  └─ Variant (Base + Salt + Formulation → 약가 매칭)
       └─ Strength (함량별 개별 코드)
```

#### 1-5. 매칭 전략 변경 (2-Pass Lookup)

기존: `normalize(raw)` → 마스터 완전 일치 (1-pass)

신규:
```
Pass 1: variant_key로 마스터 완전 일치 (정확한 약가)
Pass 2: base_key로 마스터 완전 일치 (성분 동일성 확인, 약가는 대표값)
         → match_method = "base_inn_fallback"
         → 원본 variant 정보는 보존 (나중에 약가 차이 분석 가능)
```

Pass 2로 구제되는 118건(Level 1 해당분)은 정보 손실 없이 매칭된다.
단, `match_method`가 `"base_inn_fallback"`이므로 약가 정확도가 variant 매칭보다 낮다는 걸 downstream에서 구분 가능.

#### 1-7. 설계 결정 근거

**Q: 왜 기존 `normalize_ingredient_name()`을 수정하지 않나?**
A: 기존 함수는 33,194건의 정상 매칭을 담당. 이걸 건드리면 regression 위험. decomposer는 별도 모듈로 분리하고, unmatched에 대해서만 2-Pass를 적용한다.

**Q: `ref_normalization_map`은 왜 필요한가?**
A: 마스터에 `besilate`로 등재된 성분을 MFDS가 `besylate`로 표기. 단순 삭제 시 둘 다 base_inn만 남아서 염 차이 정보가 사라짐. 표준명 치환이면 `besilate`로 통일되어 variant_key가 마스터와 일치할 수 있다.

**Q: Level 4 (첫 단어 매칭) 불허인데 구제 불가 642건은?**
A: 코드로 해결 불가. 마스터 갱신(연 1회) 또는 HIRA 체계 밖 약물이므로 `NOT_FOUND`로 유지. 무리한 퍼지매칭보다 정직한 미매칭이 데이터 품질에 이롭다.

  1. Strength: 괄호 내 (as BASE DOSE), 괄호 내 순수 %, 후행 %, 선행 %
  2. Formulation: 괄호 내 (micronized 등) + 선행 prefix (liposomal, pegylated) + 후행 longest-match
  3. Salt: SALT_FORMS + REF_NORMALIZATION_MAP 표준명 치환, base가 비면 rollback
  4. Base INN: 잔여 텍스트 소문자 정규화

#### 2-2. `ingredient_bridge.py` 2-Pass Lookup 통합

- Method 1 (기존): `normalize_ingredient_name()` → 마스터 완전 일치 — **1,910건 변동 없음**
- Method 2 (신규): Decomposition 2-Pass
  - Pass 1: `variant_key` (Base+Salt+Form) → 마스터 완전 일치 — **10건 구제**
  - Pass 2: `base_key` (Base INN) fallback — **79건 구제**
- Method 3 (기존): ATC fallback — 19건
- `_resolve_codes()` 공통화 + 이름 일치 우선순위 + decomposed 경로 오매칭 가드

## 메모

- 분해(Decomposition) 방식의 핵심 원칙: **정보는 이동하되 삭제되지 않는다**. 원본 텍스트의 모든 토큰이 4개 컬럼 중 하나에 반드시 귀속되어야 한다.
- `variant_key`는 마스터의 `일반명코드 + 제형구분코드` 조합과 대응. 마스터가 이미 `제형`과 `함량` 컬럼을 갖고 있으므로, 분해 결과와 마스터를 multi-column JOIN할 수 있는 가능성이 열린다.
- decomposed 경로에서 오매칭 가드가 중요: 복합주사제(아미노산 수액 등)에서 개별 성분명이 엉뚱한 코드로 매핑되는 구조적 문제 존재. 이름 비교로 차단.
- 기존 1,910건 normalized 매칭은 전혀 건드리지 않음 — regression 위험 제로.
- OMOP vocab 갱신 주기: RxNorm 월 1회, Athena 분기 1회. 우리 사전 재검증은 연 1회면 충분 (염/제형 표준명은 거의 불변).
