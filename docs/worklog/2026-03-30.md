## 미해결 과제: 도메인 지식 검증 문제

현재 `shared.py`의 도메인 지식(급여율, PA 기준, 규제 소요기간 등)은 **프롬프트에 하드코딩된 정적 텍스트**다. RAG도 아니고 DB 조회도 아니라서, 정책이 바뀌면 프롬프트가 **틀린 정보를 LLM에 주입**하게 된다.

### 현황 분석

```
현재 흐름:
  shared.py 하드코딩 ("산정특례 5%") → 시스템 프롬프트에 삽입 → LLM이 참고하여 출력 생성

문제:
  1. 산정특례율이 바뀌면? → 코드 배포 전까지 틀린 정보 주입
  2. GIFT 기준이 바뀌면? → 마찬가지
  3. 약물별 산정특례 해당 여부는? → 현재는 LLM이 추정 (hallucination 위험)
```

drug-briefing 4.1.0은 이미 `copay_scenario_text` 등을 **Python이 사전계산해서 `[FACT DATA]`로 주입**하는 구조를 갖고 있어 이 문제가 덜하지만, stream-briefing은 아직 약물별 급여 팩트를 주입하지 않고 있다.

### 개선 방향 4가지

#### 방향 1: 하드코딩 유지 + 출처/기준일 명시 (즉시 적용 가능)

```python
# shared.py 현재
DOMAIN_KNOWLEDGE_REIMBURSEMENT = """## 급여 도메인 지식
| 구분 | 본인부담률 | 비고 |
| 암환자 산정특례 | 5% | 중증질환 등록 필요 |
"""

# 개선 후
DOMAIN_KNOWLEDGE_REIMBURSEMENT = """## 급여 도메인 지식 (기준: 2026-01 건강보험심사평가원 고시)
| 구분 | 본인부담률 | 비고 | 근거 |
| 암환자 산정특례 | 5% | 중증질환 등록 필요 | 국민건강보험법 시행령 제19조 |
※ 이 테이블은 정적 참고자료이다. 약물별 실제 적용 여부는 [FACT DATA]의 copay_exemption 필드를 따르라.
"""
```

- **장점**: 변경 최소, LLM이 "이건 정적 참고일 뿐"이라는 걸 인지
- **단점**: 여전히 수동 업데이트 필요
- **적용 시점**: 즉시

#### 방향 2: Python 사전계산으로 이관 (Fact/Insight 원칙 정합)

stream-briefing의 `_extract_drug_intel()`이 약물별로 급여 팩트를 계산해서 `[FACT DATA]`에 주입:

```python
# _extract_drug_intel 확장
def _extract_drug_intel(drug: dict, max_fields: int = 14) -> dict:
    intel = {...}  # 기존 14필드

    # 급여 팩트 계산 (Python이 판단 → LLM은 참고만)
    mfds = drug.get("mfds_data") or {}
    hira = drug.get("hira_data") or {}

    if not mfds.get("approval_status") or mfds["approval_status"] == "미허가":
        intel["reimbursement_fact"] = "국내 미허가 — 급여 적용 불가, 처방 자체가 법적 제한"
        intel["access_routes"] = "KODC 긴급도입(전액 부담) / 제약사 EAP / 국내 임상시험 참여"
    elif hira.get("status") == "reimbursed":
        price = hira.get("price", 0)
        intel["reimbursement_fact"] = f"HIRA 급여 등재, 상한가 {price:,}원"
        if price > 1_000_000:
            intel["reimbursement_fact"] += " — 사전심사(PA) 대상"
    else:
        intel["reimbursement_fact"] = "MFDS 허가 완료, HIRA 비급여"

    # 산정특례 해당 여부 (적응증 기반)
    areas = drug.get("therapeutic_areas", [])
    if any(a in ["oncology", "hematology"] for a in areas):
        intel["copay_exemption"] = "암환자 산정특례(5%) 대상 가능"
    elif drug.get("designations") and "orphan" in drug.get("designations", []):
        intel["copay_exemption"] = "희귀질환 산정특례(10%) 대상 가능"

    return intel
```

- **장점**: Fact/Insight 원칙 완전 정합, LLM에게 팩트 판단을 맡기지 않음
- **단점**: 모든 약물에 `hira_data`, `mfds_data`가 있어야 함 (현재 스트림 파이프라인에서 수집 범위 확인 필요)
- **의존성**: `regscan/stream/base.py`의 `StreamResult.drugs_found` 구조에 hira_data 포함 여부
- **적용 시점**: 스트림 파이프라인이 hira/mfds 데이터를 수집하고 있다면 바로, 아니면 수집 확장 후

#### 방향 3: RAG — HIRA 고시/식약처 규정 실시간 검색

```
흐름:
  1. 브리핑 생성 시 약물의 적응증 추출
  2. HIRA 급여 고시 DB에서 해당 적응증의 급여 기준 검색
  3. 검색 결과를 [FACT DATA]에 주입
  4. LLM은 검색된 급여 기준을 기반으로 분석
```

구현 방식:
- **데이터 소스**: HIRA 의약품 급여 목록 (건강보험심사평가원 공개 데이터), 식약처 의약품통합정보시스템
- **검색 방식**: 약물 INN → HIRA DB 조회 → 급여 코드, 상한가, 급여 기준(적응증 제한) 추출
- **저장**: SQLite/PostgreSQL에 정기 동기화 (daily scanner에 추가) 또는 API 실시간 호출
- **주입 위치**: `_extract_drug_intel()` 내부에서 DB 조회 → `reimbursement_detail` 필드로 주입

```python
# 예시 흐름
async def _extract_drug_intel_with_rag(drug: dict) -> dict:
    intel = _extract_drug_intel(drug)  # 기존 14필드

    # HIRA DB에서 실시간 급여 정보 조회
    inn = drug.get("inn", "")
    hira_info = await db_loader.query_hira_reimbursement(inn)
    if hira_info:
        intel["hira_reimbursement"] = {
            "status": hira_info.status,           # "급여" / "비급여" / "삭제"
            "price": hira_info.price,              # 상한가
            "indications": hira_info.indications,  # 급여 인정 적응증 목록
            "pa_required": hira_info.price > 1_000_000,
            "last_updated": hira_info.updated_at,  # 고시 변경일
        }

    # 식약처 허가 정보 조회
    mfds_info = await db_loader.query_mfds_approval(inn)
    if mfds_info:
        intel["mfds_approval"] = {
            "status": mfds_info.status,
            "date": mfds_info.approval_date,
            "brand_name": mfds_info.brand_name,
        }

    return intel
```

- **장점**: 항상 최신 정보, 약물별 정확한 급여 기준
- **단점**: 인프라 무거움 (DB 동기화, API 비용), HIRA 공개 데이터 형식 파싱 필요
- **의존성**: `regscan/db/loader.py`에 HIRA/MFDS 조회 함수 추가, daily scanner에 동기화 로직
- **적용 시점**: DB 스키마에 이미 `hira_reimbursements` 테이블이 있으므로, 데이터만 채우면 가능

#### 방향 4: 도메인 지식을 "참고 맥락"으로 격하 + 팩트 우선 규칙

시스템 프롬프트에서 도메인 지식의 역할을 명확히 제한:

```python
DOMAIN_KNOWLEDGE_REIMBURSEMENT = """## 급여 도메인 지식 (일반 참고용 — [FACT DATA]가 항상 우선)

아래는 한국 건강보험 제도의 **일반적 기준**이다.
약물별 실제 급여 상태·본인부담률은 반드시 [FACT DATA]의 값을 따르라.
[FACT DATA]에 급여 정보가 없으면 "급여 정보 데이터 부족"으로 표기하라.
이 테이블의 수치를 [FACT DATA] 없이 단독으로 인용하지 마라.

| 구분 | 본인부담률 | 비고 |
|------|-----------|------|
| 일반 급여 | 30% | 외래 기준 |
...
"""
```

- **장점**: 하드코딩 유지하면서 hallucination 위험 최소화, 즉시 적용 가능
- **단점**: LLM이 급여 관련 구체적 수치를 쓰지 못하는 경우 발생 (데이터 부족 시)
- **적용 시점**: 즉시

### 권장 로드맵

| 단계 | 작업 | 시점 |
|------|------|------|
| **즉시** | 방향 4 (도메인 지식 격하 + 팩트 우선 규칙) + 방향 1 (출처/기준일 명시) | 이번 주 |
| **단기** | 방향 2 (`_extract_drug_intel` 확장 — mfds/hira 팩트 계산 주입) | 다음 스프린트 |
| **중기** | 방향 3 (HIRA DB RAG — `hira_reimbursements` 테이블 활용) | DB에 데이터 채운 후 |

방향 2와 3은 독립이 아니라 **2 → 3 순서**. 2에서 Python 사전계산 구조를 만들고, 3에서 그 데이터 소스를 하드코딩 → DB 조회로 교체하는 것.
