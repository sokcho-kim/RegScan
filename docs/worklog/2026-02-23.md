### 4. 기사 팩트 에러 3건 구조적 수정

49건 재생성 후 팩트 체크에서 발견된 3가지 구조적 결함을 Python 사전 계산으로 해결.

#### Fix 1: 산정특례 시나리오 매핑 (medclaim 할루시네이션 방지)

**문제**: 비만약(semaglutide)에 암환자 산정특례 5% 시나리오가 삽입됨. 프롬프트가 모든 약물에 산정특례를 요구하고, few-shot이 전부 항암제 → LLM이 모든 약에 산정특례 적용.

**수정**:
- `publish_articles.py`: `COPAY_EXEMPTION_MAP` + `_get_copay_exemption()` 함수 추가
  - oncology → 암환자 산정특례(5%), rare_disease/희귀 → 희귀질환(10%), 그 외 → `null`
  - impact 객체에 `_copay_exemption` 주입
- `llm_generator.py`: `_prepare_drug_data()`에 `copay_exemption` 필드 추가
- `prompts.py`: "copay_exemption이 null이면 산정특례 절대 포함 금지" 규칙 추가, few-shot 예시에 `copay_exemption` 필드 추가

**검증 결과**:

| 약물 | therapeutic_area | copay_exemption | medclaim 산정특례 | 판정 |
|------|-----------------|-----------------|-------------------|------|
| semaglutide | metabolic | `null` | **없음** | ✅ 수정됨 |
| cabozantinib | oncology | 암환자 5% | 암환자 5% | ✅ |
| zanidatamab | oncology | 암환자 5% | 암환자 5% | ✅ |
| setmelanotide | metabolic+희귀 | 희귀질환 10% | 희귀질환 10% | ✅ |

#### Fix 2: 경쟁약 필터링 정밀화 (cabozantinib ↔ semaglutide 오매칭 방지)

**문제**: `CABOZANTINIB`의 areas = `"oncology,metabolic"` → metabolic 인덱스에도 등록 → semaglutide(metabolic)의 경쟁약으로 cabozantinib(항암제)이 매칭됨.

**수정**:
- `publish_articles.py`: `area_index` 구축 시 **primary area(첫 번째)만** 등록
  - 기존: 모든 area에 등록 (oncology에도 metabolic에도)
  - 변경: primary area에만 등록 (cabozantinib → oncology에만)
- 5-b 폴백 매칭에서 `comp_primary == target_primary` 조건 강화
- `prompts.py`: "없으면 당신이 아는 약물을 언급하라" → "competitors 없으면 경쟁약 비교 생략하라" 변경

**검증 결과**: semaglutide 경쟁약에 cabozantinib **미등장** ✅

#### Fix 3: 타임라인 사전 계산 (날짜/시제 오류 방지)

**문제**: EMA 승인일 2025-06-27(과거)을 "승인 예정"으로 표현. LLM이 날짜의 미래/과거를 판단하지 못함.

**수정**:
- `llm_generator.py`: `_build_approval_timeline()` 메서드 추가
  - 각 기관별 `{agency, status, date, tense}` 반환
  - tense: `"past"` / `"future"` / `"unknown"` — Python이 사전 계산
- `_prepare_drug_data()`에 `approval_timeline` 필드 주입
- `prompts.py`: "approval_timeline의 tense 필드를 반드시 따르라, 자체 날짜 판단 금지" 규칙 추가

**검증 결과**: gpt-4o-mini는 여전히 시제 오류 (아래 모델 평가에서 해결)

```python
        submissions,
        key=lambda x: x.get("submission_status_date", ""),
        reverse=True,
    )
    latest = sorted_subs[0]  # ← 가장 최근 submission만 반환
```

**영향**:
- Cabozantinib: 2025년 보충신청(sNDA)이 최신이면 그 날짜가 "FDA 승인일"로 잡힘 → 2025-10-09
- Zanidatamab/Setmelanotide/Polatuzumab: FDA API에 아예 없거나 검색 쿼리에서 누락

#### 원인 3: DB 스키마 — (drug_id, agency) UNIQUE 제약

**파일**: `regscan/db/models.py` L124-126

```python
__table_args__ = (
    Index("idx_event_agency_drug", "drug_id", "agency", unique=True),
)
```

**영향**: 하나의 약물에 대해 FDA 이벤트가 **1행만** 저장 가능. Cabozantinib처럼 Cometriq(MTC) + Cabometyx(RCC/HCC) 등 여러 승인이 있는 약물은 최신 1건만 남음.

#### 원인 4: FDA API 검색 누락

**파일**: `regscan/ingest/fda.py`

- `search_by_generic_name()`, `search_by_substance_name()` 등 여러 메서드가 있으나, INN 정규화 불일치로 일부 약물이 검색 안 됨
- 예: "zanidatamab-hrii" (USAN 접미사 포함) vs "zanidatamab" (INN)

#### 수정 방향

| 원인 | 수정 방향 | 파일 |
|------|----------|------|

### 10. 기사 파이프라인 V4 구현 — 팩트/인사이트 분리 + Jinja2 + 툴콜링

팩트체크에서 반복 확인된 LLM hallucination 문제(날짜·금액·상태 오기)의 근본 해결을 위해 V4 아키텍처 구현.

#### V4 아키텍처 개요

```
V3: DB → _prepare_drug_data() → LLM(팩트+인사이트 동시 생성) → HTML
V4: DB → FactComputer(Python) → 사전 계산 팩트 → LLM(인사이트만) → Jinja2(팩트+인사이트 결합) → HTML
```

**핵심**: LLM이 날짜·금액·승인 상태를 직접 서술하지 않음. Python이 100% 사전 계산 → LLM은 분석 텍스트만.

#### 수정 파일 3개

| 파일 | 변경 내용 |
|------|----------|
| `regscan/report/prompts.py` | `SYSTEM_PROMPT` → `SYSTEM_PROMPT_V3` (롤백용), `SYSTEM_PROMPT_V4` 신규 (팩트/인사이트 분리 원칙, 5 Critical Rules), `BRIEFING_REPORT_PROMPT_V4` 신규 (6필드 JSON: headline, subtitle, key_points, global_insight_text, domestic_insight_text, medclaim_action_text), V3 호환 별칭 유지 |
| `regscan/report/llm_generator.py` | FactComputer 메서드 7개 (`_compute_status_text`, `_compute_d_day_text`, `_compute_copay_scenario_text`, `_compute_approval_summary_table`, `_compute_cost_scenario_table`, `_compute_valid_competitors`, `_prepare_drug_data_v4`), OpenAI 툴콜링 2개 (`get_regulatory_status`, `calculate_patient_cost`), `_call_llm_v4()` (툴콜링 루프), `generate_v4()` |
| `regscan/scripts/publish_articles.py` | `ARTICLE_TEMPLATE_V4` (Jinja2 — 팩트 영역=Python 생성 테이블, 인사이트 영역=LLM 텍스트), `generate_article_html_v4()` (facts+insights → Jinja2 렌더링 + abbr/outlink/competitor 후처리), `--v4` CLI 플래그, JSON에 `_pipeline_version` 메타 |

#### V4 사전 계산 팩트 필드 (FactComputer)

```python
{
  "d_day_text": "글로벌 승인 후 460일 경과 (국내 허가 신청 예상 구간)",
  "fda_status_text": "승인 완료 (2024-11-20)",
  "ema_status_text": "승인 완료 (2025-06-27)",
  "mfds_status_text": "미허가 (not_approved)",
  "copay_scenario_text": "향후 급여 등재 시 암환자 산정특례(5%) 적용 가능.",
  "valid_competitors": [{"inn": "Glofitamab", "domestic_status": "미허가"}],
  "approval_summary_table": "| 기관 | 상태 | 날짜 |\n...",
  "cost_scenario_table": "| 시나리오 | 본인부담률 | 비고 |\n..."
}
```

#### 툴콜링 설계 (OpenAI Function Calling)

| Tool | 용도 | 파라미터 |
|------|------|---------|
| `get_regulatory_status` | 기관별 승인 상태/날짜 조회 | inn, agency(fda/ema/mfds) |
| `calculate_patient_cost` | 시나리오별 본인부담금 계산 | drug_price, scenario(general/cancer_special/rare_special/out_of_pocket) |

LLM이 tool_calls 요청 → Python 즉시 계산 → 결과 반환 → LLM 최종 JSON (최대 3라운드).
Anthropic/Gemini는 기존 직접 호출 방식 유지.

#### 마이그레이션 전략

- V3 코드 **완전 보존** (별칭으로 기존 import 호환)
- `--v4` 플래그로 V4 활성화, 기본은 V3
- JSON에 `_pipeline_version: "v3"/"v4"` 기록
