### 1. 스냅샷 CLI: `regscan/scripts/snapshot_articles.py` (신규)

현재 `output/briefings/*.json`을 버전 라벨로 복사해 보관하는 CLI 도구.

**기능:**

| 명령 | 동작 |
|------|------|
| `--name "pre-fix"` | 현재 JSON을 `snapshots/{name}/`으로 복사 |
| `--from-commit fded439 --name "pre-v4"` | git 커밋에서 JSON 추출 → 스냅샷 저장 |
| `--list` | 기존 스냅샷 목록 출력 |

**자동 스냅샷 (`take_auto_snapshot`):**
- `publish_articles.py` 실행 완료 시 자동 호출
- 날짜+버전 기반 이름: `2026-02-25_v4`, 중복 시 `_2`, `_3` 접미
- 프롬프트 전문 저장: `_prompts.txt` (SYSTEM_PROMPT + BRIEFING_REPORT_PROMPT)
- 메타데이터: `_snapshot_meta.json` (이름, 시간, 파일수, 파이프라인 버전, auto 플래그)

**스냅샷 디렉터리 구조:**
```
output/briefings/snapshots/
├── pre-v4/                    # git commit fded439에서 추출
│   ├── _snapshot_meta.json
│   ├── POLATUZUMAB_VEDOTIN.json
│   └── ...
├── 2026-02-25_v4/             # 자동 스냅샷
│   ├── _snapshot_meta.json
│   ├── _prompts.txt           # 프롬프트 전문
│   ├── POLATUZUMAB_VEDOTIN.json
│   └── ...
```

### 2. 비교 HTML 생성기: `regscan/scripts/compare_articles.py` (신규)

두 스냅샷을 비교하는 자체 완결형 HTML(Tailwind CSS + 인라인 JS + 데이터 임베딩)을 생성.

```bash
python -m regscan.scripts.compare_articles --a "pre-v4" --b "2026-02-25_v4"
# → output/briefings/compare.html
```

**HTML 뷰어 기능:**
- 약물 드롭다운 선택기 (A/B 존재 여부 표시)
- 사이드-바이-사이드 레이아웃: Headline → Subtitle → Key Points → V4 Facts(테이블) → Global → Domestic → MedClaim
- Pipeline version / Score 배지
- 인자 없이 실행 시 대화형 스냅샷 선택

**품질 지표 자동 체크 (JS 내장):**

| 지표 | 체크 방법 |
|------|----------|
| 반복 문구 | "판매권자 부재", "시장성 판단 보류" 등 패턴 카운트 |
| 금지 표현 | "혁명적", "획기적", "사실상 제도권 밖" 등 10개 패턴 |
| 숫자 훅 | headline/global 첫 문장에 숫자 존재 여부 |
| MOA 연쇄 | "→" 패턴 2회 이상 |
| 한계점 서술 | "다만", "CI", "p-value" 등 키워드 |
| 글자수 | 전체/글로벌/국내/청구 섹션별 길이 |

### Phase C-2: 가격 스펙트럼 시스템 구현

HIRA 원본 데이터(30,418건 급여)에서 `class_no`(약효분류코드) × original/generic 세그먼트별 **백분위 가격 스펙트럼**을 사전 계산하여 V4 기사에 주입하는 시스템을 구축.

#### 배경: v3 원안 변경

| 항목 | v3 원안 | 변경 후 |
|------|---------|---------|
| 분류 키 | therapeutic_areas (EMA) | **class_no** (HIRA 원본, 120개, 미분류 0%) |
| 데이터 소스 | DB 353건 | **HIRA 원본 30,418건** |
| 통계 방식 | 단순 평균 | **백분위 스펙트럼** (P25/P50/P75/P90) |
| 세그먼트 | 없음 | **original/generic 분리** (`동일 의약품` 필드) |

데이터 조사 결과 단순 평균이 무의미함을 확인 (항암제 421: 1,000원 제네릭과 3.6억 CAR-T 공존). Plan B(사전 계산 테이블)로 설계.

#### 구현

1. **`regscan/report/price_stats.py`** (신규, 531줄)
   - `CLASS_NO_NAMES`: KD 약효분류코드 120개 매핑 + `Unknown Class ({code})` fallback
   - `rebuild_price_stats()`: HIRA JSON → `hira_price_stats` 테이블 전체 재구축 (130행)
   - `get_price_spectrum(class_no, segment)`: 통계 조회 (P25/P50/P75/P90/max)
   - `compute_drug_position(price, class_no)`: 약물의 백분위 위치 계산 ("P65" 등)
   - `check_and_rebuild_if_needed()`: SHA-256 해시 비교 → 변경 시에만 재구축
   - `ingredient_code → class_no` 역인덱스 캐시 (JSON 영속화)
   - `therapeutic_areas → class_no` fallback (미급여약용)

2. **`regscan/db/models.py`** — `HiraPriceStatsDB` 테이블 추가
   - PK: `(class_no, segment)`, 최대 240행
   - 필드: count, min_price, p25, p50_median, p75, p90, max_price, source_hash

3. **`regscan/report/llm_generator.py`** — `_compute_price_spectrum()` 추가
   - 급여약: `hira_code` → ingredient_code → class_no → 스펙트럼 + drug_position
   - 미급여약: `therapeutic_areas` → class_no fallback → 스펙트럼만
   - `_prepare_drug_data_v4()`에 `price_spectrum` 필드 자동 주입

4. **`regscan/report/prompts.py`** — 가격 스펙트럼 해석 규칙 추가
   - P50/P90 인용 필수, 단순 평균 금지
   - max_price가 P50의 100배 이상이면 "CAR-T 등 초고가 치료제 포함" 맥락 추가

5. **`regscan/scripts/publish_articles.py`** — 파이프라인 시작 시 `check_and_rebuild_if_needed()` 자동 호출

#### 검증

| 테스트 | 결과 |
|--------|------|
| rebuild (30,418건 → 130행) | 정상 |
| 421/original (항악성종양제) | 976건, P50=90,896원, P90=525,296원 |
| Unknown fallback (`999`) | `"Unknown Class (999)"` |
| drug_position (150,000원 in 421) | P65 |
| 해시 동일 파일 재실행 | 스킵 정상 |
| 급여약 (ingredient_code 매칭) | 스펙트럼 + drug_position |
| 미급여약 (therapeutic_areas fallback) | 스펙트럼만 |
| 분류 없음 | None (미주입) |
