# 콘텐츠 품질 개선 설계 — 기사 + 팩트카드 이중 출력

> 작성일: 2026-05-12
> 문제: 기사 품질이 낮다. 프롬프트가 아니라 데이터가 얇은 게 근본 원인.
> 목표: (1) 기사 품질을 올리고 (2) 팩트카드를 추가 콘텐츠로 뽑는다.

---

## 1. 현재 문제 진단

### 기자에게 들어가는 데이터

```
시그널 제목 (30자) + 요약 (100-300자) + 메타데이터 (날짜, URL, 소스)
+ enrichment (FDA label 300자, 승인이력, e약은요 300자)
```

이걸로 700-1000자 기사를 쓰라니까 **입력보다 출력이 길어서** LLM이 빈 공간을 채운다:
- "~로 읽힌다" (용어 풀이)
- "~가 변수가 될 수 있다" (당연한 전망)
- 무관한 보도자료 항목까지 나열 (분량 채우기)
- AI 특유 추상 표현 ("~축에서", "~측면")

### enrichment.py 현황

| 소스 | 엔리칭 | 추가 데이터 |
|------|--------|------------|
| PMDA_APPROVAL | FDA label + 승인이력 + CT.gov + HIRA 급여 | 풍부 |
| MFDS_PRESS | e약은요 (효능/용법/부작용) | 보통 |
| KIPRIS_PATENT | 키워드→INN→FDA label | 보통 |
| ASSEMBLY_BILL | Playwright로 제안이유 크롤링 | 풍부 (있으면) |
| GNW_PRESS | 원문 HTML 본문 추출 | 풍부 |
| KHIDI_GLOBAL_INFO | 국내 크로스레퍼런스만 | **얇음** |
| MFDS_SAFETY_LETTER | FDA label | 보통 |
| KHIDI_PHARMA_NEWS | 없음 (그대로 통과) | **없음** |

**핵심:** 엔리칭이 약물 중심(FDA/e약은요)으로만 되어 있고, 규제/법령/산업 시그널의 엔리칭이 빈약.

---

## 2. 해결 방향: 두 트랙

### Track A: 기사 품질 — 데이터를 두텁게

**원칙: 입력 > 출력. 기자가 "버릴 데이터"가 있어야 좋은 기사가 나온다.**

#### A-1. 원문 전문 fetch

현재 시그널에는 제목+요약만 있다. 원문을 가져와야 한다.

| 소스 | 원문 접근 방법 | 예상 데이터량 |
|------|--------------|-------------|
| MFDS 보도자료 | 이미 PDF 추출 구현됨 (`mfds_press.py`) | 1-5페이지/건 |
| KHIDI 게시판 | 상세 페이지 HTML fetch (board/view URL 있음) | 500-2000자/건 |
| 국회 법안 | Playwright로 제안이유 (이미 구현됨) | 1500자/건 |
| GNW 보도자료 | 이미 원문 fetch 구현됨 | 2000자+/건 |
| PMDA | RSS 원문 링크 | 미구현 |

**즉시 가능:** KHIDI 상세 페이지 fetch. `KHIDIIngestor.fetch_detail()` 패턴이 이미 `khidi.py`에 있음. `khidi_global.py`에도 동일 패턴 적용하면 됨.

#### A-2. 과거 관련 기사/이벤트 조회

같은 약물/규제에 대해 과거에 어떤 일이 있었는지 맥락 제공.

```
예: "급여 상한금액표 개정" 기사 작성 시
→ DB에서 해당 약물의 과거 급여 이력 조회
→ "2025년 1월 2차 치료 한정 급여 → 이번에 1차 확대"
→ 기자가 "기존과 차이" 블록을 실데이터로 채움
```

구현 방법:
- `regscan.db`에서 동일 INN/고시번호 과거 이벤트 조회
- enrichment에서 `_fetch_related_history(drug_or_regulation_id)` 추가

#### A-3. 규제 시그널 전용 엔리칭

현재 `_enrich_global_with_domestic()`이 국내 크로스레퍼런스만 함.
규제 기사에 필요한 엔리칭:

| 필요 데이터 | 소스 | 구현 |
|------------|------|------|
| 고시/법령 원문 | KHIDI 상세 페이지 | fetch_detail() |
| 변경 전 규정 | RegScan DB 과거 이벤트 | SQL 조회 |
| 해외 유사 규제 선례 | NICE/PMDA/FDA 과거 데이터 | DB 조회 |
| 영향 받는 약물 목록 | 고시번호 → 약물 매핑 (HIRA) | 데이터레이크 |

### Track B: 팩트카드 — 구조화된 정보 카드

기사와 별개로, 모든 시그널에서 **구조화된 팩트카드**를 뽑는다.

#### B-1. 팩트카드 스키마

```json
{
  "card_type": "regulation_change | drug_approval | patent_filing | market_data",
  "title": "급여 상한금액표 일부개정 고시 제2026-91호",
  "effective_date": "2026-05-01",
  "source": "보건복지부",
  "source_url": "https://...",
  
  // regulation_change 전용
  "before": "A약제 2차 치료 이후 급여",
  "after": "A약제 1차 치료까지 확대",
  "affected_drugs": ["약물A INN"],
  "affected_entities": ["종합병원 약제팀", "보험심사 담당자"],
  "deadline": "의견조회 5/20 종료",
  
  // drug_approval 전용  
  "drug_name": "Otarmeni",
  "inn": "...",
  "indication": "...",
  "agency": "FDA",
  "approval_date": "2026-04-28",
  "korea_status": "미허가",
  
  // 공통
  "tags": ["급여확대", "항암제", "고시개정"],
  "impact_score": 8
}
```

#### B-2. 팩트카드 생성 파이프라인

```
시그널 + enrichment
    → LLM 팩트 추출 (구조화 JSON)
    → 검증 (필수 필드 체크)
    → DB 저장 (fact_cards 테이블)
    → 프론트 API (/api/v1/fact-cards)
```

**기사 파이프라인과 독립.** 기사는 주 1-2회, 팩트카드는 매일.

#### B-3. 프론트 활용

- MedicinePage: 약물 카드에 관련 팩트카드 연결
- 규제 트래커: 팩트카드 타임라인으로 규제 변경 추적
- 대시보드: 최신 팩트카드 피드

---

## 3. 기사 ↔ 팩트카드 관계

```
데이터 수집 (13+ 소스)
    │
    ├── 시그널 추출
    │       │
    │       ├── 팩트카드 생성 (매일, 모든 시그널)
    │       │       → DB 저장
    │       │       → 프론트 카드형 표출
    │       │
    │       └── 기사 생성 (주 1-2회, 편집장 선별)
    │               → 팩트카드를 참조 데이터로 활용
    │               → 원문 + 과거 이력 + 팩트카드 = 두터운 입력
    │               → 기사 품질 ↑
    │
    └── enrichment (원문 fetch, 과거 이력, 크로스레퍼런스)
```

**핵심:** 팩트카드가 기사의 입력 데이터가 된다. 팩트카드에 "before/after", "affected_drugs" 같은 구조화 정보가 있으면, 기자가 뻔한 소리 대신 **실데이터로 기사를 채울 수 있다.**

## 4. 구현 우선순위

### Phase 1: 기사 데이터 두텁게 (즉시)

1. KHIDI 상세 페이지 fetch — `khidi_global.py`에 `fetch_detail()` 추가
2. MFDS 보도자료 원문을 기자에게 전달 (이미 수집되지만 시그널에 포함 안 됨)
3. `format_for_prompt()`에서 content 필드 truncation 완화 (300→1000자)
4. 기자 프롬프트: 뻔한소리/용어풀이/무관정보 가드레일 강화

### Phase 2: 팩트카드 파이프라인 (이번 주)

1. 팩트카드 스키마 확정
2. LLM 팩트 추출 프롬프트 작성
3. DB 테이블 생성 (fact_cards)
4. 생성 파이프라인 (시그널 → 팩트카드 JSON)
5. API 엔드포인트 (/api/v1/fact-cards)

### Phase 3: 기사가 팩트카드를 참조 (다음 주)

1. 기자 입력에 관련 팩트카드 주입
2. 과거 이벤트 히스토리 조회 추가
3. 기사-팩트카드 연결 (article_id ↔ fact_card_ids)

### Phase 4: 프론트 표출

1. MedicinePage에 팩트카드 연동
2. 규제 트래커 페이지 (팩트카드 타임라인)
3. 대시보드 Feed에 팩트카드 통합

---

## 5. 기대 효과

| 현재 | 개선 후 |
|------|---------|
| 기사만 있음 (주 1-2회) | 기사 + 팩트카드 (매일) |
| 시그널 요약 300자로 기사 작성 | 원문 + 팩트카드 + 히스토리로 작성 |
| LLM이 빈 공간을 뻔한 소리로 채움 | 버릴 데이터가 있어서 선택적 기술 |
| 프론트에 줄 콘텐츠 부족 | 팩트카드로 일일 콘텐츠 확보 |
| 기사 중심 (긴 글) | 카드형(스캔) + 기사(딥다이브) 이중 구조 |
