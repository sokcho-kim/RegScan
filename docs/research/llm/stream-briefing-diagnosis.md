## 2. 4구성요소 진단

### SYSTEM_PROMPT (L26-62)

| 요소 | 존재 | 품질 | 문제점 |
|------|------|------|--------|
| **Instruction** | O | 중 | "원칙" 6개가 instruction 역할. 그러나 대부분이 "하지마라" 형태 (금지 위주). |
| **Context** | △ | 하 | 오늘 날짜 `{today}`만 주입. Persona가 1문장("규제 인텔리전스 전문 분석가")으로 빈약. drug-briefing은 "메드클레임 인사이트 수석 의약 전문기자, 10년 경력" 수준의 구체적 정체성. |
| **Input Data** | - | - | SYSTEM_PROMPT에는 없음 (개별 프롬프트에서 처리) — 이것은 정상. |
| **Output Indicator** | △ | 하 | "순수 JSON만 출력, 한글" — 스키마 없이 지시만. 개별 프롬프트에서 스키마가 있으나 SYSTEM에도 포맷 규칙이 있어야 함. |

### 스트림별 프롬프트 (THERAPEUTIC/INNOVATION/EXTERNAL/UNIFIED)

| 요소 | 존재 | 품질 | 문제점 |
|------|------|------|--------|
| **Instruction** | O | 중 | `[TASK]` 섹션으로 분리됨. 그러나 "작성하라"만 있고, 분석 프레임(어떤 관점으로 사고할지)이 없음. |
| **Context** | O | 중 | `[FACT DATA]` 섹션으로 분리됨. 구분자 사용은 좋음. |
| **Input Data** | O | 상 | `{drug_details}`, `{signals}` 등 14필드 인텔리전스 주입. V3에서 크게 개선됨. |
| **Output Indicator** | O | 중 | JSON 스키마 있음. 그러나 필드별 길이 제한이 없는 곳이 많고(trend_analysis "3-5줄"은 모호), 타입 명시 없음. |

**4구성요소 구분자 사용**: `[FACT DATA]`, `[TASK]`, `## 원칙`, `## 시간 추론 규칙` — 기본은 갖춤. 그러나 영역 간 경계가 덜 명확함 (원칙과 톤 예시가 섞여 있음).

## 3. 기법별 Gap 분석

### 3-1. Few-Shot: 완전 입출력 쌍 ❌

**현재 (stream-briefing 1.2.0)**:
```
### GOOD (이렇게 써라)
"headline": "FDA, KRAS G12C 이중억제제 sotorasib 병용요법 2026-02-14 승인 완료"
"why_it_matters": "국내 비소세포폐암 2차 치료 시장(연 4,200명)에서..."
```
→ **톤 예시 2개뿐.** 입력이 없고, 출력의 일부 필드만 표시. Min et al. (2022)에 따르면 이것은 Few-shot이 아님.

**목표 (drug-briefing 4.1.0 수준)**:
```
입력: {"inn": "PEMBROLIZUMAB", "fda": {"approved": true, "date": "2014-09-04"}, ...}
출력: {"headline": "...", "key_points": [...], "global_section": "...", ...} (전체 필드)
```
→ 3가지 시나리오(승인 완료 / 장기 미허가 / 희귀질환) 각 1쌍씩, **완전한 JSON 입출력**.

**Gap**: GOOD/BAD 톤 예시 → 최소 2쌍의 완전 입출력으로 교체 필요. 치료영역 1쌍 + 혁신 1쌍이 최소 목표.

### 3-2. Chain-of-Thought: 추론 과정 시연 ❌

**현재**:
```
## 시간 추론 규칙 (필수)
- 승인일/허가일 < 오늘({today}) → "승인 완료", "허가됨" (과거형)
- 승인일/허가일 > 오늘({today}) → "승인 예정", "심사 중" (미래형)
```
→ **규칙만 나열.** LLM이 규칙을 읽고 올바르게 적용한다는 보장이 없음.

**목표**:
```
예시: fda_date=2025-03-28, 오늘=2026-03-30
→ 2025-03-28 < 2026-03-30 → 과거 → "2025-03-28 FDA 승인 완료"

예시: pdufa_date=2026-09-15, 오늘=2026-03-30
→ 2026-09-15 > 2026-03-30 → 미래 → "2026-09-15 PDUFA 예정"
```
→ 실제 날짜로 **추론 과정을 시연**. LLM이 패턴을 모방.

**Gap**: 규칙 3줄 → CoT 시연 2-3개로 교체/보강.

### 3-3. 금지표현 대안표 ❌

**현재**: 없음. GOOD/BAD 톤 예시가 있지만 2개뿐이고, 체계적 대안표가 아님.

**drug-briefing 4.1.0에는 5개 금지/대안 쌍**이 명시되어 있음:
| 금지 | 대안 |
|------|------|
| "~할 것으로 예상된다" | 경과일·패턴 기반 구체적 해석 |
| "모니터링이 필요하다" | 구체적 시기·대상 명시 |
| "주목된다" | 구체적 이유 명시 |
| "새로운 치료 옵션을 제시할 수 있다" | 구체적 포지셔닝 |
| "지속적인 관심이 필요하다" | 구체적 액션 |

**Gap**: 전면 추가 필요. 스트림 브리핑 특성상 "혁명적", "획기적", "유망한" 등도 추가.

### 3-4. 생애주기별 분기 ❌

**현재**: 모든 약물에 동일 프롬프트 적용. 미허가·심사중·승인완료 약물이 같은 형태로 처리됨.

**drug-briefing 4.1.0에는 4가지 분기**:
- `[글로벌 승인 O / 국내 미허가 / 기간 오래됨]` → 미충족 수요 + 긴급도입
- `[국내 급여 삭제/미급여]` → 환자 경제적 부담 + 비급여 리스크
- `[혁신 신약 PRIME/Breakthrough]` → 기존 표준치료 한계 + 선점 경쟁
- `[국내 허가+급여 완료]` → 적응증별 급여 기준 + 처방 최적화

**Gap**: 스트림에 맞게 재설계 필요. 최소 3분기: 미허가 / 심사중 / 승인완료.

### 3-5. Persona 구체성 ❌

**현재**: "제약·바이오 산업의 규제 인텔리전스 전문 분석가" — 1문장, 추상적.

**drug-briefing 4.1.0**: "메드클레임 인사이트의 수석 의약 전문기자, 10년간 FDA/EMA/MFDS 규제 동향과 건강보험 급여 정책을 취재한 베테랑 기자" — 구체적 정체성, 경력, 전문 분야.

**Gap**: Persona를 3-5문장으로 확장. 분석가의 경력·전문성·독자 타겟 명시.

### 3-6. Prompt Chaining ⚠️ (검토 필요)

**현재**: 통합 브리핑(UNIFIED_BRIEFING_PROMPT)에 6가지 독립 작업이 한 프롬프트에 있음:
1. 3개 스트림 요약 읽기
2. 교차 신호 분석
3. top 5 약물 선정 + 각 약물별 action
4. risk alerts 도출
5. opportunities 도출
6. tomorrow_watch

→ **5개 이상 독립 작업** 기준 초과.

**가이드 원칙**: 5개 이상 독립 작업이면 Prompt Chaining으로 분할하라.

**Gap**: 2단계 분할 검토.
- Step 1: 교차 신호 추출 + top 5 선정
- Step 2: Step 1 결과로 executive summary + risk/opportunity 도출

### 3-7. DO 중심 지시 ⚠️ (부분 적용)

**현재 "하지마라" 빈도 분석**:

| 지시 | 유형 |
|------|------|
| "허위 생성 금지" | DON'T |
| "절대 만들지 마라" | DON'T |
| "절대로 추정하거나 생성하지 마라" | DON'T |
| "코드블록/마크다운 금지" | DON'T |
| "단순 반복·요약하지 마라" | DON'T |
| "BLUF — 첫 문장에서 답을 제시하라" | DO |
| "시간 추론 규칙을 반드시 준수하라" | DO |
| "숫자는 구체적으로" | DO |

→ DON'T 5개 vs DO 3개. **금지가 행동보다 많음.**

**가이드 원칙**: "하지마라"는 모델이 무엇을 해야 하는지 알려주지 않는다. 원하는 행동을 명시하라.

**Gap**: "허위 생성 금지" → "날짜는 [FACT DATA]에 있는 값만 사용하라. 없으면 '데이터 부족'으로 표기하라" 형태로 전환.

### 3-8. 도메인 지식 주입 ❌

**현재**: SYSTEM_PROMPT에 도메인 지식 없음.

**drug-briefing 4.1.0에는**:
- 급여 도메인 지식 테이블 (본인부담률 5종)
- 규제 프로세스 도메인 지식 (FDA→MFDS 소요기간, MFDS→HIRA 소요기간, GIFT 설명)

**Gap**: 스트림별로 필요한 도메인 지식 주입:
- 치료영역: 급여/약가 프로세스, 산정특례 기준
- 혁신: PRIME/Breakthrough/Orphan 지정의 의미와 심사 단축 효과
- 외부시그널: 임상실패 시 기존 처방 영향, medRxiv의 신뢰도 수준

### 3-9. 분석 프레임 (CoT 내부 추론용) ❌

**현재**: `[TASK]`에 "작성하라"만 있음. LLM이 어떤 사고 과정을 거쳐야 하는지 안내 없음.

**drug-briefing 4.1.0에는 5단계 분석 프레임**:
1. 약물 정체성 파악
2. 뉴스 앵글 결정
3. 경쟁 지형 분석
4. 타임라인 해석
5. 메드클레임 시나리오

→ "아래 순서로 사고하되, 출력은 JSON만 작성하세요"

**Gap**: 스트림별 분석 프레임 설계 필요.

## 5. NAVER D2 / 배민 기준 대비

### NAVER D2 — 코드에서 프롬프트 분리

| 기준 | 현재 | 평가 |
|------|------|------|
| 프롬프트를 코드에 하드코딩하지 마라 | `briefing.py` 파일 상단에 문자열 상수로 하드코딩 | ❌ |
| 버전 관리 도구에서 관리 | Git으로만 추적, Langfuse/Vitess 등 프롬프트 관리 도구 없음 | ❌ |
| 배포 없이 프롬프트 변경 가능 | 코드 배포 필요 | ❌ |

→ **현재 RegScan 규모에서는 하드코딩이 합리적**. 다만 2.0.0에서 프롬프트가 대폭 커지면 별도 파일(`prompts/stream_therapeutic.py` 등) 분리를 고려.

### 배민 AI Platform 2.0 — LLMOps 4컴포넌트

| 컴포넌트 | 배민 | RegScan 현재 | 평가 |
|----------|------|-------------|------|
| GenAI Studio (Langfuse) | 프롬프트 버전 + Observability | Git 기반 수동 관리 | 규모 대비 적정 |
| GenAI SDK (LiteLLM) | 모델 통합 + Fallback | `_call_llm()` 자체 구현 (OpenAI→Gemini→Anthropic) | ✅ 동등 |
| GenAI API Gateway | OpenAI 호환 + Routing | 없음 (단일 서비스) | 불필요 |
| GenAI Labs | 실험 + Golden Dataset | 없음 | ❌ 필요 |

→ **Golden Dataset + 체계적 실험 시스템이 가장 큰 Gap**. 최소한 "정답 브리핑"을 수동 작성하고 LLM 출력과 비교하는 Reference-based 평가를 도입해야 함.

## 6. drug-briefing 4.1.0 vs stream-briefing 1.2.0 직접 비교

| 기법 | drug-briefing 4.1.0 | stream-briefing 1.2.0 | Gap |
|------|---------------------|----------------------|-----|
| **Persona** | 수석 의약 전문기자, 10년 경력, 구체적 독자 타겟 | "규제 인텔리전스 전문 분석가" 1문장 | 재설계 |
| **Few-Shot** | 3쌍 완전 입출력 (PEMBROLIZUMAB, POLATUZUMAB, SELADELPAR) | GOOD/BAD 톤 예시 2개 (headline, why_it_matters만) | 전면 추가 |
| **CoT** | 5단계 분석 프레임 (약물 정체성→뉴스 앵글→경쟁→타임라인→메드클레임) | 없음 | 전면 추가 |
| **금지표현 대안표** | 5쌍 (예상된다→경과일 기반, 모니터링→구체적 시기...) | 없음 | 전면 추가 |
| **생애주기 분기** | 4분기 (미허가/미급여/혁신/급여완료) | 없음 | 전면 추가 |
| **도메인 지식** | 급여 5종 테이블 + 규제 프로세스 4항목 + GIFT | 없음 | 전면 추가 |
| **시제 규칙** | approval_timeline.tense 필드 기반 (Python이 판단) | 날짜 비교 규칙 (LLM이 판단) | **stream이 더 리스크** — Python 사전 판단으로 전환 검토 |
| **산정특례 규칙** | copay_exemption null이면 생략, 적응증 불일치 금지 | 없음 | 해당 시 추가 |
| **출력 필드 수** | 10+ (headline, subtitle, key_points, global_heading/section, domestic_heading/section, medclaim_section) | 5-7 (headline, key_takeaway, top_drugs, trend_analysis, action_items) | stream은 요약이므로 적정 |
| **Fact/Insight 분리** | Python 사전계산 → LLM 인사이트만 | `[FACT DATA]` 주입 → LLM 분석 | ✅ 동일 패턴 |

## 8. 예상 토큰 비용 변화

| 버전 | 프롬프트 토큰 (추정) | 비용 영향 |
|------|---------------------|----------|
| **1.2.0 현재** | SYSTEM ~600 + 스트림별 ~400 = ~1,000/call | 기준 |
| **2.0.0 예상** | SYSTEM ~1,200 + 스트림별 ~1,500 (Few-shot 포함) = ~2,700/call | **~2.7x** |
| **drug-briefing 4.1.0 참고** | ~4,000/call | 3쌍 Few-shot이 대부분 |

Few-shot 추가로 토큰이 2-3배 증가하지만, 일관성·품질 향상이 비용을 정당화함 (drug-briefing 4.1.0에서 검증됨).

## 9. 현재 프롬프트의 강점 (유지할 것)

1.2.0에서 잘 되어 있는 부분:

1. **[FACT DATA] / [TASK] 구분자**: 입력과 지시가 명확히 분리됨
2. **JSON 스키마 직접 명시**: 출력 형태가 명확
3. **시간추론 규칙 존재**: 규칙 자체는 정확 (시연만 없음)
4. **4관점 분석 가이드** (why_it_matters): "경쟁구도/급여가격/환자규모/처방변화" — 이 구조는 우수
5. **통합 브리핑의 교차 분석 지시**: "같은 약물이 여러 스트림에 등장하면 신호가 강하다" — 이 통찰은 유지
6. **key_takeaway 필수 규칙**: 누락 방지 명시
7. **`_extract_drug_intel` 14필드**: 풍부한 컨텍스트 주입 (V3의 핵심 개선)
