# Claude 헛발 방지 변경 요약 — 2026-04-30

> 범위: 2026-04-29~2026-04-30에 수행한 기사 생성 가드레일 보강
> 주의: 저장소에 다른 미커밋 변경이 있으므로, 이 문서는 아래 4개 파일에 대한 변경만 요약한다.

- `regscan/article/guardrails.py`
- `regscan/article/pipeline.py`
- `docs/article-pipeline.md`
- `docs/review/2026-04-29_article_fact_check.md`

---

## 배경

2026-04-29 생성 기사 검토에서 아래 문제가 확인됐다.

- 법안 기사에서 공개 자료와 맞지 않는 날짜/조문 조합 사용
- 제목은 `2건`인데 본문은 더 많은 사례를 전개하는 범위 불일치
- 사실 문장과 해석 문장이 섞여 과장처럼 읽히는 문장 존재
- 사용자에게 보고된 기사 요약과 실제 생성 파일 내용 불일치

핵심 문제는 "프롬프트가 약해서"가 아니라, **LLM 출력에 대해 코드 레벨 검증이 부족했다는 점**이다.

---

## 변경 요약

### 1. `regscan/article/guardrails.py`

신규 추가:

- `validate_article_grounding(story, article, original_data)`
- `_extract_full_dates(text)`

추가한 검증:

- `regulation` 기사에서 기사 본문에 등장한 날짜가 원본 데이터에 없는 경우 감지
- `regulation` 기사에서 기사 본문에 등장한 조문 번호가 원본 데이터에 없는 경우 감지
- 제목에 `N건`이 들어갔는데 본문에서 `TA966`, `TA976` 같은 사례 ID가 그보다 많이 나오면 감지

의도:

- 클로드가 법안 기사에서 날짜와 조문을 조합해 "그럴듯하게" 쓰는 문제 차단
- 제목과 본문이 서로 다른 기사를 말하는 문제 차단

한계:

- 현재 날짜 추출은 문자열 정규식 기반이다.
- Windows 콘솔 한글 인코딩 문제 때문에 로컬 샘플 출력만으로는 완전 검증이 어려웠다.
- 따라서 이 함수는 "완전한 팩트체크"가 아니라 "자주 발생하는 헛발 탐지"에 초점을 둔다.

---

### 2. `regscan/article/pipeline.py`

구조 변경:

- 파이프라인 설명을 `v3 (2-Pass)`로 정리
- 기존 `4-Agent` 설명을 `5-Agent` 구조로 명시
- `Structure-Validator`를 독립 에이전트로 추가

프롬프트 보강:

- 편집장 프롬프트에 `사실 고정 규칙` 추가
  - 날짜/조문/법안 단계는 원본 데이터에 있는 값만 사용
  - 제목에 `N건`을 쓰면 본문도 그 범위 안에서만 전개
  - 법안 기사에서 조문 번호는 원본 데이터에 실제 존재할 때만 사용
- 기자 프롬프트에 금지 규칙 추가
  - 원본 데이터에 없는 날짜/조문 단정 금지
  - 제목 `N건`보다 많은 대표 사례 전개 금지
  - `[국내 크로스레퍼런스]` 데이터 사용 의무 강화

실행 로직 변경:

- `agent_structure_validator()`를 2nd pass에 연결
- `slot_results`를 최종 기사 메타에 저장
- A등급 기사에서 `structural_info`, `domestic_angle`, `evidence` 핵심 슬롯이 `NOT_FOUND` 2개 이상이면 `A → B` 다운그레이드
- `validate_article_grounding()` 결과가 있으면:
  - `A → B`, `B → C` 다운그레이드
  - `_grounding_issues`를 최종 기사에 기록

정리하면:

- 예전에는 "프롬프트로 잘 써라"에 가까웠고,
- 지금은 "구조 누락"과 "근거 불일치"를 코드에서 별도로 잡는 구조로 바꿨다.

### 3. `docs/article-pipeline.md`

문서 정리:

- 파이프라인 개요를 `4-Agent Loop`에서 `5-Agent 2-Pass`로 수정
- `Structure-Validator`와 `validate_article_grounding()`를 문서에 반영
- 현재 품질 관리가 `LLM 단계`와 `코드 단계`로 나뉜다는 점을 명시

특히 추가한 내용:

- 법안 기사 날짜 불일치 검증
- 법안 기사 조문 불일치 검증
- 제목 `N건`과 본문 사례 수 불일치 검증
- 이 검증에 걸릴 경우 등급을 자동 하향한다는 정책

문서 목적:

- 현재 파이프라인이 어디까지 자동으로 막고, 어디부터는 사람이 다시 봐야 하는지 경계를 명확히 남기기 위함

신규 작성 문서.

포함 내용:

- 후속 개선안

역할:

- 이번 코드 변경이 왜 필요했는지 설명하는 근거 문서

## 기대 효과

이번 변경으로 바로 좋아지는 부분은 아래다.

1. 법안 기사에서 날짜/조문 헛발 가능성 감소
2. NICE류 기사에서 제목-본문 범위 불일치 감소
3. 구조적으로 빈약한 A등급 기사 자동 강등
4. 국내 크로스레퍼런스 무시 가능성 일부 감소

## 아직 안 막히는 것

이번 변경이 모든 문제를 해결하는 건 아니다.

- 원본 데이터 자체가 잘못 들어오면 여전히 잘못 쓸 수 있다
- 가격 정보의 1차 출처 승격은 아직 미완료
- 생성 후 사용자에게 보여주는 요약 텍스트와 실제 파일의 일치 검증은 아직 코드화하지 않았다
- 한글 인코딩 문제 때문에 일부 문자열 기반 검증은 Windows 콘솔 테스트만으로 완전 검증되지 않았다

즉, 지금 상태는 **"클로드가 자주 하던 헛발 몇 가지를 코드로 바로 막기 시작한 1차 방어선"**이다.
